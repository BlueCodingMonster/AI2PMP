# 需求池精简设计

## 1. 背景与目标

AI2PmP 第一阶段只建设需求池、计划、任务三个可独立管理的环节，环节之间的数据关联均为可选。本文仅定义需求池模块，字段以 `doc/需求池.xlsx` 为业务依据，并增加明确的创建时间字段。

本次改造目标：

- 需求可以不依赖项目、计划或任务独立创建和维护。
- 删除现有需求模型中未被业务表使用的字段，降低录入和理解成本。
- 保留轻量评审状态，不建设复杂审批流。
- 统一表单、列表、详情、筛选、校验和数据库模型的字段口径。

计划和任务模块不在本文实施范围内。需求与计划、任务的既有可选关联继续保留，后续分别设计其精简方案。

## 2. 数据模型

### 2.1 保留及新增字段

| 字段 | 规则 | 录入方式 |
| --- | --- | --- |
| 内部主键 | 数据库内部使用，不在业务界面展示 | 系统生成 |
| 需求编号 | 从 `XQ0001` 开始全局递增；超过四位后自然扩展，如 `XQ10000`；全局唯一 | 系统生成 |
| 需求名称 | 必填，去除首尾空格，长度 2 至 100 个字符 | 人工填写 |
| 归属产品线 | 可为空，不影响需求独立创建 | 人工选择 |
| 优先级 | 高、中、低 | 人工选择 |
| 需求来源 | 产品规划、客户反馈、公司内部需求、市场需求 | 人工选择 |
| 客户名称或需求方 | 可为空，保存客户、部门或提出人等自由文本 | 人工填写 |
| 提出时间 | 可为空 | 人工选择 |
| 创建人 | 当前登录用户 | 系统生成 |
| 创建时间 | 首次保存需求的服务器时间，后续编辑不得修改 | 系统生成 |
| 需求状态 | 待评审、评审中、已评审、已驳回、已排期、已完成 | 人工选择；新建默认待评审 |
| 评审通过时间 | 可为空；需求状态为已评审时必填 | 人工选择 |

业务编号使用独立的自增序号生成。数据库内部技术主键不改成业务编号，以免影响既有外键关系。

### 2.2 删除字段

以下字段及其专属数据库索引、表单控件、列表列、筛选项、详情展示、查询逻辑和写入逻辑全部删除：

- 需求类型
- 详细描述
- 业务价值评分
- 技术复杂度评分
- 预估工期
- 关联项目及对应外键
- 负责人及对应外键
- 验收标准

这是数据库层面的直接删除，不保留隐藏兼容字段。执行迁移前需要确认并接受这些字段中现有数据会被永久删除。

### 2.3 可选关联边界

- `归属产品线` 是需求自身的可选分类字段。
- 需求不要求关联项目。
- 计划或计划事项可以选择关联需求，也可以不关联。
- 任务可以选择关联需求，也可以不关联。
- 删除需求时，计划和任务侧的可选外键按 `SetNull` 处理，不级联删除计划或任务。

## 3. 状态设计

需求状态只包含：

1. 待评审
2. 评审中
3. 已评审
4. 已驳回
5. 已排期
6. 已完成

状态之间不设置顺序约束，登录且具备现有需求编辑权限的用户可以直接切换到任意状态。

保存校验规则：

- 新建需求默认状态为待评审。
- 状态为已评审时，评审通过时间必须填写，否则拒绝保存并提示“请选择评审通过时间”。
- 其他状态下，评审通过时间允许为空，也允许保留此前人工填写的日期。
- 状态变更不得自动写入、覆盖或清空评审通过时间。

旧状态迁移映射：

| 旧状态 | 新状态 |
| --- | --- |
| 草稿 | 待评审 |
| 评审中 | 评审中 |
| 已批准 | 已评审 |
| 已拒绝 | 已驳回 |
| 已排期 | 已排期 |
| 进行中 | 已排期 |
| 已完成 | 已完成 |
| 已延期 | 待评审 |

旧数据从“已批准”迁移为“已评审”时，如果原记录没有评审通过时间，迁移允许暂时为空；该记录下一次被编辑保存时必须补填日期。这一迁移例外只用于兼容历史数据，不放宽新建和编辑校验。

## 4. 表单设计

新建和编辑表单只呈现以下业务信息：

- 需求编号：新建时显示“保存后自动生成”，编辑时只读展示。
- 需求状态：新建默认为待评审，允许直接选择其他状态。
- 需求名称。
- 归属产品线。
- 优先级。
- 需求来源。
- 客户名称或需求方。
- 提出时间。
- 创建人：只读。
- 创建时间：新建时显示“首次保存时生成”，编辑时只读展示。
- 评审通过时间：人工日期选择。

当用户把状态选为已评审时，评审通过时间的标签显示必填标记。前端在提交前提示，服务端使用同一规则再次校验，避免绕过页面提交非法数据。

## 5. 列表与筛选

列表按以下顺序展示：

1. 需求编号
2. 需求名称
3. 归属产品线
4. 优先级
5. 需求来源
6. 客户名称或需求方
7. 提出时间
8. 创建时间
9. 创建人
10. 需求状态
11. 评审通过时间
12. 操作

列表支持：

- 产品线筛选。
- 状态筛选。
- 优先级筛选。
- 按需求编号、需求名称、客户名称或需求方进行关键字搜索。
- 默认按创建时间倒序排列。

列表不再加载项目、负责人、需求类型、价值评分或复杂度数据。

## 6. 详情页

详情页展示与列表、表单相同的 11 个业务字段。已经存在的计划和任务可选关联可以在详情页作为只读信息展示；没有关联时不显示空面板。

详情页不再显示已删除字段。评论能力不属于本次字段精简范围，不在本次改造中新增、扩展或删除。

## 7. 数据迁移

迁移顺序：

1. 增加全局自增序号和人工评审通过时间字段。
2. 为现有需求分配连续序号，保证业务编号唯一。
3. 按状态映射表转换旧状态。
4. 将现有来源映射到四个新来源；旧“内部建议”归入“公司内部需求”，旧“市场分析”归入“市场需求”，旧“技术债务”归入“公司内部需求”。
5. 将旧优先级“紧急”归入“高”。
6. 删除旧字段、外键、索引和不再使用的枚举类型。
7. 重新生成 Prisma Client，并验证数据库结构。

迁移必须保持需求与产品线、计划、任务之间仍然有效的可选关联。

## 8. 错误处理

- 未登录用户创建、编辑、删除或变更状态时返回现有未登录错误。
- 需求名称为空或长度非法时拒绝保存。
- 状态为已评审但评审通过时间为空时拒绝保存。
- 需求编号由数据库序号保证唯一；并发创建不使用“查询最大值再加一”的方式。
- 产品线标识不存在时拒绝保存，不静默写入空关联。
- 数据库写入失败时返回统一的用户可读错误，并在服务端记录原始错误。

## 9. 测试与验收

采用测试先行方式覆盖以下行为：

- 新建需求只提交精简字段即可成功。
- 新建需求自动获得连续且唯一的 `XQ` 编号、创建人和创建时间。
- 需求不选择产品线也可以保存。
- 六种状态均可直接选择。
- 已评审但未填写评审通过时间时，前后端均拒绝保存。
- 已评审并填写评审通过时间时可以保存。
- 其他状态不填写评审通过时间时可以保存。
- 列表列顺序、筛选条件、搜索范围和默认排序符合本文定义。
- 详情页不再读取或展示删除字段。
- 迁移后原有需求数量不减少，产品线及计划、任务可选关联不丢失。
- Prisma 校验、项目 lint、构建以及需求池关键页面冒烟测试通过。

## 10. 非目标

本次不处理：

- 计划表单和计划模型精简。
- 任务表单和任务模型精简。
- 审批角色、审批意见、会签或多级审批。
- 自动状态流转。
- 需求与计划、任务的强制关联。
- 需求评论能力重构。
- 批量导入 `doc/需求池.xlsx`。
