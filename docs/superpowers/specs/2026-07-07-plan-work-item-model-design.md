# 计划与工作项模型设计

## 设计目的

AI2PmP 应让产品线小组用尽量简单的方式管理产品研发和项目执行。核心业务模型为：

```text
产品线小组
  -> 计划
  -> 工作项
  -> 任务
```

产品线、产品、模块、版本、项目等信息用于辅助说明工作归属，不应要求用户在记录实际工作前先判断并创建多层管理对象。现行年度工作计划表中的每一行，应优先理解为一个年度工作项，而不是一个独立项目。

## 核心对象

### 产品线

产品线负责业务和产品归属，用于承载年度计划、工作项、产品/平台、模块/板块和版本等信息。现行表格中的「产品线名称」应优先对应系统中的产品线。

### 团队

团队负责人员和组织关系，用于维护小组成员、组长、产品负责人、开发负责人、临时借调和人员负载等信息。

产品线管理和团队管理应在用户入口上分开：

```text
产品线管理 = 管业务归属、产品、模块、版本、年度工作项
团队管理 = 管人员、角色、借调、负载
```

如果底层暂时仍使用 `ProductLineTeam` 承载两类信息，前端也应在交互和文案上区分「产品线管理」和「团队管理」，避免用户把产品线误解为纯组织小组。

### 计划

计划表示一个有明确时间范围的管理周期。

支持的正式计划层级：

- 年度计划
- 季度计划
- 月度计划

计划之间可以形成父子关系。年度计划可以汇总和承接全年工作项，季度计划用于里程碑视图，月度计划用于执行视图。半年度不作为正式必建计划对象，只作为统计视图：

```text
上半年 = Q1 + Q2
下半年 = Q3 + Q4
```

### 计划归属与管理视图

所有正式计划和执行记录都应归属于产品线小组。年度计划、季度计划、月度计划、工作项、阶段安排、任务和计划外工作，都以产品线小组作为责任边界。

```text
产品线小组
  -> 年度计划
  -> 季度计划
  -> 月度计划
  -> 工作项
  -> 阶段安排
  -> 任务
  -> 计划外工作
```

管理者需要的是跨产品线小组的整体视图，而不是另建一套独立计划。系统应基于产品线小组计划数据汇总形成管理视图：

```text
管理者年度计划视图 = 所有产品线小组年度计划汇总
管理者季度计划视图 = 所有产品线小组季度计划和阶段安排汇总
管理者月度计划视图 = 所有产品线小组月度执行、任务和计划外工作汇总
```

因此，计划数据的创建、维护、执行和责任追踪发生在产品线小组内；跨组查看、筛选、统计、预警和复盘发生在管理视图中。管理视图不改变计划归属，只负责汇总展示。

### 计划变动性

计划不是一次创建后永久有效的静态对象。年度计划、季度计划、月度计划之间存在拆解和汇总关系，但下级计划可能因为上级计划调整、资源变化、紧急事项、外部要求或管理口径变化而作废、替换或新增。

计划应支持以下生命周期状态：

```text
草稿
已发布
执行中
已调整
已作废
已完成
已归档
```

基于上层计划拆解出的下级计划，应记录来源关系，但不能被设计成不可变的强绑定关系：

```text
下级计划
  -> 来源计划：可选
  -> 来源工作项：可选
  -> 拆解批次：可选
  -> 替代计划：可选
  -> 作废原因：可选
  -> 调整说明：可选
```

规则：

- 年度计划拆解出季度计划或月度计划后，下级计划可以继续调整。
- 上级计划变更时，系统应提示受影响的下级计划，但不自动覆盖已经执行中的下级计划。
- 下级计划作废时，不删除历史数据，应标记为已作废，并记录作废原因、作废时间和操作人。
- 作废计划不再计入当前计划完成率，但应保留在历史追溯和复盘中。
- 如果需要替换作废计划，应新建替代计划，并保留新旧计划之间的替代关系。
- 如果某个维度临时增加新计划，例如按产品、模块、专项、客户、负责人或风险事项新增计划，应作为同一产品线小组下的独立计划，并通过关联字段连接到年度、季度或月度计划。

因此，系统中的计划关系应优先采用弱关联，而不是强制树形结构。推荐模型是：

```text
年度计划
  -> 可拆解出季度计划、月度计划
  -> 可关联专项计划、产品计划、模块计划

季度计划
  -> 可来源于年度计划
  -> 可拆解出月度计划
  -> 可被调整、作废、替换

月度计划
  -> 可来源于年度计划或季度计划
  -> 可直接新增计划项
  -> 可承接计划外工作
  -> 可被调整、作废、替换
```

管理视图应区分当前有效计划和历史变动记录。默认看当前有效计划，同时允许切换查看已作废、已替换、已调整的计划链路。

### 工作项

工作项是承载实际工作的统一对象。它可以表示：

- 需求
- 版本目标
- 项目事项
- 运维支撑
- 市场支撑
- 技术调研
- 缺陷或紧急修复
- 其他临时工作

工作项可选关联以下归属信息：

- 产品线
- 产品
- 模块
- 关联版本
- 版本名称
- 关联项目
- 来源
- 优先级
- 负责人
- 时间范围
- 状态
- 进度

### 工作来源与任务类型

产品线小组的工作来源不只有平台研发规划，也包括本地化部署和项目交付。系统应统一用工作项承载实际工作，但需要区分工作来源和任务类型，避免把不同管理口径混在一起。

推荐工作来源：

```text
平台研发规划
本地化项目交付
计划外工作
```

平台研发规划类工作包括：

- 产品规划
- 版本研发
- 模块优化
- 技术研究
- 场景验证
- 知识产权
- 技术调研
- 缺陷修复

本地化项目交付类工作包括：

- 本地化部署
- 客户项目交付
- 定制化开发
- 数据迁移
- 系统对接
- 联调测试
- 现场实施
- 验收支持
- 交付后问题处理

技术研究、场景验证、知识产权三类任务只属于平台研发规划类计划，不属于项目交付计划。项目交付计划应聚焦交付过程本身，不强制包含技术研究、场景验证、知识产权任务。

项目交付过程中如果产生可沉淀的平台能力、技术研究方向、场景验证成果、专利、软著或标准线索，由产品线小组线下判断和处理。系统不要求在项目交付计划中体现这些线索，也不建立从项目交付工作项到平台研发专项任务的强制转换或关联流程。

平台版本研发和项目定制化研发使用同一套内部研发管理流程。两者的差异在于工作来源和统计口径不同，不在于研发流程不同：

```text
平台版本研发
  -> 工作来源 = 平台研发规划
  -> 执行流程 = 内部研发管理流程

项目定制化研发
  -> 工作来源 = 本地化项目交付
  -> 执行流程 = 内部研发管理流程
```

内部研发管理流程包含产品、研发、测试三大主要环节，并按以下阶段推进：

```text
项目/任务启动会
需求分析
需求评审及宣讲
研发启动
需求分析及研发设计
设计文档评审及宣讲
代码编写
研发集成测试
提测
测试及问题修复
上线发布
项目/任务总结
```

系统设计上，研发流程应作为可复用流程模板，而不是只绑定平台版本研发。平台版本研发工作项和项目定制化研发工作项都可以选择该模板，并按阶段安排和任务推进。

为了降低使用复杂度，系统不应默认把完整流程节点全部平铺给用户。推荐采用三大环节的简化视图：

```text
产品环节
  -> 项目/任务启动会（关键里程碑）
  -> 需求分析
  -> 需求评审及宣讲（关键里程碑）

研发环节
  -> 研发启动（关键里程碑）
  -> 需求分析及研发设计
  -> 设计文档评审及宣讲
  -> 代码编写
  -> 研发集成测试

测试与发布环节
  -> 提测（关键里程碑）
  -> 测试及问题修复
  -> 上线发布
  -> 项目/任务总结
```

日常操作界面默认展示三大环节、四个关键里程碑和当前关键节点；完整流程节点作为模板明细，在需要细化计划、跟踪节点、查看历史时展开。这样既保留内部研发流程的完整性，又避免计划编写和日常跟踪过重。

关键里程碑包括：

- 项目/任务启动会
- 需求评审及宣讲
- 研发启动
- 提测

这些里程碑应体现在对应环节中，用于判断流程是否进入下一阶段，但不需要把所有细分节点都提升为默认可见节点。

同一个项目交付计划中，不同事项可以使用不同流程模板：

| 交付事项类型 | 推荐执行流程 |
|---|---|
| 本地化部署 | 部署准备、环境配置、数据初始化、联调、验收 |
| 定制化开发 | 内部研发管理流程 |
| 数据迁移 | 数据梳理、映射设计、迁移执行、校验、验收 |
| 系统对接 | 接口确认、联调开发、测试、上线、验收 |
| 现场实施 | 实施准备、现场配置、用户培训、问题处理、验收 |

因此，项目交付计划不需要包含技术研究、场景验证、知识产权三类专项任务，但必须支持其中的定制化开发事项按内部研发管理流程进行过程管理。定制化研发的统计口径仍属于本地化项目交付，不转为平台研发规划。

专项任务的三类基础元数据保持一致：

```text
专项任务类型：技术研究 / 场景验证 / 知识产权
知识产权子类型：专利 / 软著 / 标准，可选
编号
任务名称
目标
完成时间
承担机构或人员
状态
进度
```

附件中的表格可映射为专项任务模板：

| 表格字段 | 系统字段 |
|---|---|
| 编号 | 专项任务编号 |
| 任务名称 | 专项任务名称 |
| 目标 | 专项目标 |
| 完成时间 | 计划完成时间 |
| 承担机构或人员 | 责任机构或责任人 |

做计划时不强制先在产品线管理中维护版本。版本是工作项的可选归属，不是计划录入的前置条件。

工作项支持三种版本处理方式：

```text
1. 关联已有版本
2. 直接填写版本名称
3. 不填写版本
```

推荐字段：

```text
关联版本ID：可选
版本名称文本：可选
```

规则：

- 如果选择已有版本，使用关联版本ID。
- 如果版本库还未维护，可以直接填写版本名称文本。
- 如果是调研、支撑、运维、市场材料、临时事项，可以不填版本。
- 后续可以提供「同步创建到产品线版本库」动作，但不能作为创建计划的前置条件。

### 阶段安排

阶段安排是工作项下面的计划节奏单元，用于表达某个工作项在某个阶段、某个周期由谁负责推进。

现行年度计划表中的「产品规划计划 / 产品负责人」和「开发、测试计划 / 开发负责人」不应固化为工作项的两个固定字段，而应作为两个默认阶段模板：

```text
工作项：大陆通平台 V1.9.6

阶段安排：
1. 产品规划
   计划时间：2024年12月
   负责人：张传梅

2. 开发测试
   计划时间：2025年1月
   负责人：马永辉
```

阶段安排的字段包括：

- 阶段名称
- 计划时间
- 负责人
- 状态
- 说明

阶段名称不限定为「产品规划」和「开发测试」。默认可按产品、研发、测试与发布三大环节展示，必要时再展开到以下细分阶段：

- 项目/任务启动会
- 需求调研
- 需求分析
- 需求评审及宣讲
- 研发启动
- 需求分析及研发设计
- 产品设计
- 设计文档评审及宣讲
- 开发
- 代码编写
- 研发集成测试
- 测试
- 提测
- 测试及问题修复
- 上线
- 上线发布
- 验收
- 项目/任务总结
- 运维支撑
- 市场材料

其中，项目/任务启动会、需求评审及宣讲、研发启动、提测是内部研发管理流程的关键里程碑，应在环节视图中默认体现。

季度和月度归属应由阶段安排的计划时间推导，而不是在工作项上额外维护一组固定月份字段。

### 任务

任务是工作项或阶段安排下面的具体执行单元。任务状态和任务进度是工作项进度的主要依据。

常见任务示例：

- 需求澄清
- 产品设计
- 前端开发
- 后端开发
- 联调
- 测试
- 发布
- 文档整理

## 年度、季度、月度关系

正式计划层级保持三层：

```text
年度计划
  -> 季度计划
  -> 月度计划
```

年度计划是录入主表，承载年度工作项。季度计划是里程碑视图，月度计划是执行视图。季度和月度视图由工作项下的阶段安排自动归类。

示例：

```text
年度计划：2025 年度产品工作计划
工作项：数典知识库 V1.1.0

阶段安排：
- 产品规划：2024年12月，负责人张传梅
- 开发测试：2025年1月，负责人王云飞
```

系统推导：

```text
产品规划阶段 -> 2024年12月计划 / 2024 Q4
开发测试阶段 -> 2025年1月计划 / 2025 Q1
```

如果阶段计划时间为「第二季度」，则归入对应季度视图；后续可以在季度或月度执行时继续细化到具体月份和任务。

## 计划内工作流程

计划内工作按以下链路管理：

```text
产品线小组
  -> 年度计划
  -> 工作项
  -> 阶段安排
  -> 任务
```

示例：

```text
产品线小组：数字化建设组
年度计划：2025 年度产品工作计划
工作项：考勤管理 v1.2.0
阶段安排：产品规划 2025年2月，开发测试 第二季度
任务：需求评审、班次规则调整、前端修改、后端修改、测试、上线
```

年度计划用于记录全年任务总表。季度计划用于按阶段计划时间汇总里程碑。月度计划用于承接具体月份内需要推进的阶段安排和任务。

## 计划外工作流程

计划外工作出现时，不应强制塞入月度计划。它应先进入产品线小组下的计划外工作池。

```text
产品线小组
  -> 计划外工作池
  -> 工作项
  -> 任务
```

计划外工作示例：

- 客户临时需求
- 线上问题
- 紧急缺陷
- 市场支撑
- 运维保障
- 领导临时安排
- 技术调查

每个计划外工作项需要标记后续处理方式：

- 不纳入计划，仅记录
- 补入当前月度计划
- 转入下个月度计划
- 关联到已有工作项

这样既能记录真实工作量，又不会污染原始月度计划。

## 进度规则

进度应从任务向上汇总：

```text
任务进度
  -> 阶段安排进度
  -> 工作项进度
  -> 月度计划进度
  -> 季度计划进度
  -> 年度计划进度
```

除非有明确的人工修正原因，否则不建议在高层对象上手动维护进度。默认进度应由任务状态计算得出。

推荐汇总规则：

- 阶段安排进度由阶段下任务计算。
- 工作项进度由其下阶段安排和任务计算。
- 月度计划进度由归入该月的阶段安排和任务计算。
- 季度计划进度由归入该季度的阶段安排和月度计划计算。
- 年度计划进度由年度工作项、季度视图和月度执行情况汇总。
- 计划外工作应与计划完成率分开统计。
- 已作废计划不计入当前有效计划完成率。
- 已调整和已替换计划应在复盘中展示调整前后差异。

## 统计与复盘

月度复盘应分开展示计划内工作和计划外工作：

```text
月度计划完成率：75%
计划外工作项：12 项
计划外工时：86 小时
计划外工作占比：28%
受计划外工作影响延期的计划项：3 项
```

这样可以避免把计划延期简单理解为执行失败。如果团队实际被大量紧急工作、临时支撑或线上问题占用，系统应能反映这种资源占用。

管理者整体视图应支持按年度、季度、月度切换，并能按产品线小组、产品线、负责人、阶段、计划内/计划外等维度筛选。默认展示所有产品线小组的汇总数据，同时保留下钻到单个产品线小组计划的能力。

统计与复盘应能展示计划变动情况，包括新增计划数、作废计划数、替代计划数、计划外转计划内数量、因上级计划调整导致变化的下级计划数量。这样管理者看到的不只是完成率，也能看到计划稳定性和资源扰动情况。

统计与复盘还应区分工作来源，至少展示平台研发规划、本地化项目交付、计划外工作的数量、进度和资源占比。技术研究、场景验证、知识产权三类专项任务只纳入平台研发规划统计口径，不纳入项目交付计划统计口径。

## 导航模型

主导航应保持简洁：

- 工作台
- 计划
- 工作项
- 任务
- 产品线管理
- 团队管理

产品线管理负责产品线、产品/平台、模块/板块和版本；团队管理负责人员、小组、角色、借调和负载。产品、模块和版本作为基础资料维护，在工作项表单中作为可选归属字段出现，但不作为计划录入前置条件。

## 现行年度表格映射

现行年度工作计划表可按以下规则导入或录入：

| 表格字段 | 系统对象 |
|---|---|
| 产品线名称 | 产品线小组 / 产品线 |
| 序号 | 工作项排序号 |
| 任务名称 | 工作项标题 |
| 简述 | 工作项描述 |
| 任务名称中的版本号 | 工作项版本名称文本，或关联已有版本 |
| 产品规划计划 | 阶段安排：产品规划 的计划时间 |
| 产品负责人 | 阶段安排：产品规划 的负责人 |
| 开发、测试计划 | 阶段安排：开发测试 的计划时间 |
| 开发负责人 | 阶段安排：开发测试 的负责人 |
| 备注 | 工作项备注 |

系统列表可以继续展示成现有表格样式，但底层应使用「工作项 + 阶段安排」结构，避免把产品规划和开发测试写死成唯一两种流程。

导入现行表格时，如果任务名称中包含版本号，例如「大陆通平台 V1.9.6」，系统可以先作为工作项标题和版本名称文本保存，不要求用户先在产品线管理中创建该版本。

## 设计结论

系统不建议把以下链路作为主要用户操作路径：

```text
产品线小组
  -> 产品线
  -> 产品/平台
  -> 模块/板块
  -> 版本库
  -> 项目
  -> 计划
  -> 任务
```

推荐的用户操作路径是：

```text
产品线
  -> 年度计划或计划外工作池
  -> 工作项
  -> 阶段安排
  -> 任务
```

这种设计更符合实际产品线管理过程，同时仍然保留产品、模块、版本和项目等必要上下文。
